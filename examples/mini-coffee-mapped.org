* Spec validation notes                                          :review:
What worked, what didn't, what worg needs to revise.

** What mapped cleanly
- Stages → level-1 headlines with #+TODO: state machine. ✓
- Inter-stage ordering → :DEPENDS_ON: with [[id:]] links. ✓
- Success criteria → child headlines tagged with :KIND: property. ✓
- Tool invocations → either child headline OR source block.
  /Headline-per-invocation reads better/ — gives you state tracking,
  retry semantics, and a place for :LOGBOOK: per call.
- Artifacts produced → :ARTIFACT: property. ✓
- Cost tracking → :COST_USD: property per call, summed in run metadata. ✓
- State transitions → :LOGBOOK: drawer. ✓
- Retries on FAIL → multiple FAILED siblings before a DONE. Org's
  default TODO transitions don't track this natively; we lean on
  :LOGBOOK: timestamps to reconstruct order. /Acceptable./

** What fit awkwardly
1. *Source-block "languages" we invented* — =fs-write= and the gamut
   tool names (=screenplay.parse=, =velocity.propose=, etc.) aren't
   real Babel languages. Two options:
   a. Treat every gamut tool as a /shell/ block that shells out to
      =gamut <subcommand>= (what the runtime already does internally).
      Cleaner — uses only canonical org.
   b. Define a worg-specific dispatcher: =#+BEGIN_SRC gamut :tool
      screenplay.parse=. Less canonical but more explicit about which
      tool is being called vs which CLI fronts it.
   /Recommend (a)./ Keeps the parser pure-org. The :TOOL: property
   on the parent headline already disambiguates for the scheduler.

2. *Tool arguments as :ARG_*: properties is verbose.* The transcript
   has =args={"path":"brief.md","content":"..."}= — we ended up
   flattening to =:ARG_PATH:= + =:ARG_CONTENT:=. For tools with
   nested object args this gets ugly. Option: stash full JSON in a
   single =:TOOL_ARGS:= property as a fenced string, OR use a
   =#+begin_src json= block inside the headline body.
   /Recommend the JSON source-block approach for any args > 3 keys./

3. *"Validator" vs "Tool" distinction has no native org marker.* Both
   are sub-headlines. Today we tag with :KIND: on validators and
   :TOOL: on tools, but a scheduler reading the file needs to branch
   on which property exists. Could use org /tags/ instead:
   =* DONE Brief check :validator:= vs =* DONE fs.write :tool:=.
   /Recommend tag-based discrimination — it's already canonical
   org and queryable by =worg query --tag=validator=./

4. *The "Run log" section is derived data living next to source data.*
   Org doesn't distinguish — both are just outline. The runtime would
   need to know "this section is mine, regenerate it on every write."
   Convention: a top-level headline with =:ARCHIVE:= tag, or a
   property like =:DERIVED: t=. /Recommend =:DERIVED: t= property
   per derived section./

5. *No native way to express "this stage has a budget cap that
   applies to the SUM of child tool calls."* :BUDGET_USD: on the
   parent works for the property model, but enforcement needs the
   scheduler to walk descendants and sum :COST_USD:. /Fine — that's
   not a syntax issue, it's a query/runtime issue. worg-query
   handles it./

6. *Failed-tool retries clutter the outline.* We have three FAILED
   headlines + one DONE for dialogue.tts. Reads chronologically but
   visually noisy. Alternative: collapse retries into the LOGBOOK of
   a single headline, with each attempt's args + result as a list
   entry. /Recommend the single-headline+LOGBOOK approach for
   retries — keeps the outline scannable./

** Open question for the spec
- Do we want a :TRUST_LEVEL: property on source blocks for the
  executor dispatch (sandboxed/full)? The mini-coffee case is
  full-trust (gamut CLI), but Studio's per-org agent surface will
  need the distinction. Document but don't lock in syntax yet —
  let it emerge from the second use case.
