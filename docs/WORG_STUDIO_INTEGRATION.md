# WORG ↔ Studio integration

How Studio materializes worg state into Postgres for cross-board
queries + reactive LiveView, without breaking the .org-is-canonical
contract.

> **Status:** schema shipped (`apps/studio/workhorse/lib/workhorse/db/worg/`
> + migration `20260520140000_create_worg_tables.exs`). This document
> is the design rationale + the update-flow contract that any future
> consumer (Pi extension, third-party adapter) should follow.

## Source of truth

The `.org` file is canonical. Postgres rows are a **derived index** —
fast to query, fast to subscribe to (Phoenix.Sync shapes), but
reconstructible from the .org at any time. If they ever diverge, the
.org wins.

This shape is intentional. It means:

- Authoring tools (editors, Workhorse mutations via worg-cli) can write
  the .org bytes and trust Postgres to catch up.
- Recovery from a corrupted DB is just a re-parse of every .org under
  the org.
- An offline-only consumer (workbook artifact in a browser, OSS user
  with no Studio account) can read the .org directly — no DB needed.

## Schema

Four tables. See `apps/studio/workhorse/lib/workhorse/db/worg/*.ex` for
the actual Ecto modules.

### `worg_documents`

One row per `.org` file. Slug is URL-routable + unique.

| Column | Type | Notes |
|---|---|---|
| `id` | `binary_id` PK | autogenerated |
| `slug` | `string` UNIQUE | e.g. `team/sprint-2026-w22` |
| `org_id` | `string` | from `#+ID:` keyword if present |
| `group_id` | `string` | broker-managed multi-tenancy scope; null for single-tenant |
| `r2_key` | `string` | Cloudflare R2 object key holding the .org bytes |
| `version_hash` | `string` NOT NULL | sha256 of the .org bytes; changes on every mutation |
| `title` | `string` | from `#+TITLE:` |
| `inserted_at` / `updated_at` | `utc_datetime_usec` | standard |

Indexes: `slug` (unique), `group_id`, `org_id`.

### `worg_headlines`

One row per headline. Parent_id is null for top-level headlines.

| Column | Type | Notes |
|---|---|---|
| `id` | `binary_id` PK | autogenerated DB id |
| `document_id` | `binary_id` FK → `worg_documents` | |
| `parent_id` | `binary_id` nullable | self-FK for outline tree |
| `org_id` | `string` | the `:ID:` property — stable across document edits |
| `level` | `integer` | 1, 2, 3, ... (asterisks count) |
| `title` | `string` | the heading text, sans state/priority/tags |
| `state` | `string` nullable | workflow keyword (TODO/NEXT/DOING/DONE/etc) |
| `priority` | `integer` nullable | from `[#A]`/`[#B]`/`[#C]` inline marker |
| `ordinal` | `integer` | preserves document order for stable rendering |

The `:ID:` ↔ DB primary-key split is deliberate. `:ID:` is the
worg-side stable identifier (referenced by `:BLOCKER:`, `:TRIGGER:`,
`:PARENT:`); the DB primary key is what FKs from `worg_tags` /
`worg_properties` point at. Two different identity domains.

### `worg_tags`

Many-to-one tags on headlines.

| Column | Type | Notes |
|---|---|---|
| `id` | `binary_id` PK | |
| `headline_id` | `binary_id` FK → `worg_headlines` | |
| `tag` | `string` | one row per tag |

### `worg_properties`

Many-to-one `:PROPERTIES:` drawer entries.

| Column | Type | Notes |
|---|---|---|
| `id` | `binary_id` PK | |
| `headline_id` | `binary_id` FK → `worg_headlines` | |
| `key` | `string` | uppercase (case-folded on insert per wb-qwj8.3) |
| `value` | `string` | verbatim |

## Update flow (the contract every writer follows)

Every mutation lands in a single `Ecto.Multi` so partial writes don't
leak. The Multi is owned by the writer process (the worg-agent Loop,
the worg-cli `claim`/`log`/`transition` subcommands, future Studio
LiveView editor):

```elixir
Multi.new()
|> Multi.run(:parse, fn _, _ ->
  with {:ok, new_org} <- Worg.transition_todo(prev_org, headline_id, "DONE"),
       parsed <- Worg.Parser.parse_headlines_json(new_org) do
    {:ok, %{org: new_org, parsed: parsed, hash: :crypto.hash(:sha256, new_org)}}
  end
end)
|> Multi.run(:r2_put, fn _, %{parse: %{org: bytes}} ->
  Workbooks.R2.put(doc.r2_key, bytes)
end)
|> Multi.update(:doc_version, fn %{parse: %{hash: h}} ->
  Ecto.Changeset.change(doc, version_hash: Base.encode16(h, case: :lower))
end)
|> Multi.run(:reindex, fn _, %{parse: %{parsed: headlines}} ->
  Workhorse.Worg.Sync.upsert(%{slug: doc.slug, headlines: headlines}, version_hash)
end)
|> Repo.transaction()
```

If any step fails, the transaction rolls back — Postgres stays in
the pre-mutation state. R2 doesn't roll back (it's not part of the
DB transaction), but the next reconciler tick (below) catches the
drift.

## Reconciler

A periodic job that scans `worg_documents`, fetches each `.org` from
R2, re-parses, and compares the parsed result to the in-DB rows. If
they diverge:

1. Log the divergence (with the document slug + version_hash).
2. Rebuild Postgres rows from the .org. **.org wins.**

Frequency: hourly default; on-demand via a `Mix.Task` for ops
debugging. The reconciler is best-effort — its purpose is to detect +
heal silent corruption, not to be the primary write path.

## Group-level multi-tenancy

`worg_documents.group_id` carries the broker-managed multi-tenancy
scope. Null means single-tenant / no-group. Per-org enforcement is
done at the broker boundary (`broker/worker/src/routes/groups.ts`);
this schema just carries the FK so cross-org queries return correct
slices.

For per-tenant Postgres (if we ever go there), the group_id stays —
it becomes redundant with the database the connection points at, but
keeps the cross-org query path open if both architectures coexist.

## What's intentionally NOT in this schema

- **No `body` column.** Headline body text lives in the .org. The DB
  has just enough structure to answer "what TODOs are open?" / "what's
  blocked by X?" / "show me everything tagged :stage:" — for full
  content you re-fetch from R2.
- **No `:BLOCKER:` / `:TRIGGER:` join tables.** These live in
  `worg_properties` as plain key=value rows. Query joins do their own
  parsing of the value list (`[[id:foo]] [[id:bar]]`). Adding
  dedicated tables would denormalize for a single query pattern; the
  current shape stays generic.
- **No `:LOGBOOK:` / `:RESULTS:`.** Drawer + result blocks live in
  the .org body; Postgres doesn't index them. If a future use case
  needs full-text search across logbook entries, that's a separate
  index (Postgres full-text or an external search engine), not a
  column on `worg_headlines`.
